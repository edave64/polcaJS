What?

polca uses prefix or postfix notation. That means that the operator is written before or after the operands, respectively. For example you write "4 8 +" in postfix and "+ 4 8" in prefix instead of "4 + 8" in traditional notation. Prefix notation is also known as Polish notation or Łukasiewicz notation after Jan Łukasiewicz who discovered it 1924.

Postfix notation is also known as reverse Polish notation or RPN. Programmming languages like Forth, FALSE and Factor are built upon postfix notation. This notation can be simply interpreted using the concept of a stack.

How?

Making a valid prefix or postfix expression is easy, for anyone. For this example we will use postfix, prefix is just the other way around. First, you write down the operands, like "2 5" then add write the operator after that "2 5 *" you can than use this expression as an operand itself by simple substitution, thus "6 x +" becomes "6 2 5 * +" or substitute one of i's own operands "2 5 *". You can literally read that as a sequence of operations to be made in that order.

Relations like = or > can be seen as operations that return an truth value, which are represented as 1 and 0 in polca. thus "3.5 3.5 =" returns 1, while "2 2 = 2 6 > ->" returns -1.

Why?

Traditional mathematical notation is somewhat messy, binary functions (in this case: operations that have two operands) are usually written in an infix style like "2 + 3". this, while somewhat intuitive, brings a lot of problems:

"2 + 3 * 4" most basically, is an undefined statement. you will probably not agree, since you learned that some operations have precedence over others, but that's an additional rule that isn't part of infix notation. another rule would be to just give precedence to the thing that is written first, in this case "2 + 3" would be evaluated first.

To some extend the traditional way works ok, some operations seem to be 'higher' than others, but besides that this is not really true for all binary operations (is a root higher than a logarithm? what about logical operators?), it's already breaks down when the operation isn't associative. you have to invent another rule that says that equal operation are evaluated from left to right (or the other way around).

But there aren't only binary operations, there also unary operations that you have to prefix or postfix anyway. traditional notation does it both so that even simple statements like "sin 4 !" are ambiguous, so unary operations need precedence, too!

This is normally dealt with by the use of parenthesis, which can be quite excessive if you want to express something that is not thougth of by the usual rules.

Prefix and postfix notation both are fully unambiguous, without needing any parenthesis. as demonstrated here:

2 3 4 * +
2 3 + 2 *
4 ! sin
4 sin !
